# GODO AI OS 프로젝트 인수인계 문서

## 1. 프로젝트 목적

GODO AI OS는 고도몰 기반 쇼핑몰 운영을 보조하는 AI 운영 시스템이다.

목표는 단순 챗봇이 아니라, 쇼핑몰 운영팀처럼 움직이는 다중 AI 에이전트 운영 대시보드를 만드는 것이다.

핵심 방향은 다음과 같다.

* 주문 확인
* CS 문의 분류 및 답변 초안
* 배송/송장 상태 확인
* 리뷰 답글 초안
* 재고 위험 감지
* 매출 요약
* 마케팅 캠페인 제안
* 정책/지식 기반 판단
* 승인 필요한 작업은 사람이 최종 승인

최종 철학은 “완전 자동화”가 아니라 “Human-approved AI Operations OS”이다.

즉, AI가 초안 작성, 분석, 감지, 제안을 담당하고, 위험도가 높은 실행은 운영자가 승인하는 구조다.

## 3. 전체 시스템 개념

GODO AI OS의 현재 구조는 다음과 같다.

OFFICE:
운영 현장 대시보드. 픽셀 오피스, 에이전트 작업 상태, Today’s Tasks, Approval Queue, Activity Log를 보여준다.

AGENTS:
AI 직원 명부. 각 에이전트의 역할, 지식, 스킬, 도구, 권한, 메모리를 확인한다.

BRAIN:
쇼핑몰 운영 AI들이 참조하는 공용 지식 저장소. 정책 문서, 템플릿, 리포트, 의사결정 로그, 재고 스냅샷 등을 관리한다.

STUDIO:
운영자가 앱 안에서 Brain 문서, Agent 설정, Skill, Tool, Permission Matrix를 직접 편집하는 관리자 편집실이다.

ENGINE:
작업을 Local, Cloud, Hybrid, Human Approval 중 어디로 보낼지 결정하는 AI 모델 라우터 설정 화면이다.

WORKFLOW:
START OPERATION 실행 시 각 AI 에이전트가 작업을 수행하는 Mock Workflow Engine이다.

APPROVAL:
고위험 작업은 자동 실행하지 않고 Approval Queue로 보내 사람이 승인/거절한다.

REPORT:
운영 종료 후 자동 처리 건수, 승인 대기 건수, 위험 신호, 권장 조치를 리포트 모달로 보여준다.

* draft_only
* approval_required
* manual_only

## 9. 고도몰 API 연동 방향

고도몰 API가 열려 있으면 가능할 것으로 예상되는 영역:

* 주문 조회
* 상품 조회
* 재고 조회
* 게시판 정보 조회
* 일부 게시판 등록 또는 수정

CS 에이전트가 직접 답글을 다는 구조는 기술적으로 가능할 가능성이 있다.

하지만 초기에는 반드시 다음 구조로 가야 한다.

1. 고도몰 API로 미답변 문의 조회
2. CS AI가 문의 분류
3. cs_policy.md, refund_exchange_policy.md, product_expression_rules.md 참조
4. 답변 초안 생성
5. 운영자가 Approval Queue에서 확인
6. 승인 후 API로 답글 등록
7. 결과 로그 저장

처음부터 완전 자동 답글 등록은 금지한다.
초기 권한은 draft_only 또는 approval_required가 맞다.

위험 작업:

* 환불 실행
* 가격 변경
* 고객 삭제
* 쿠폰 즉시 발급
* 대량 문자
* 상품 대량 수정

이런 작업은 manual_only 또는 approval_required로 유지해야 한다.

## 10. 중요한 보안 원칙

프론트엔드 React 코드에 API Key를 넣으면 안 된다.

실제 고도몰 API나 LLM API를 붙일 때는 다음 구조가 필요하다.

React Frontend
→ Vercel Function 또는 Node Backend
→ 환경변수에서 API Key 사용
→ 고도몰 API 호출
→ 개인정보 마스킹
→ 프론트로 필요한 데이터만 반환

API Key는 GitHub에 올리면 안 된다.
Vercel Environment Variables 또는 백엔드 환경변수로 관리해야 한다.
