## A. 한눈에 보는 결론

RC-1은 "새 기능 추가"가 아니라 **같은 것을 부르는 이름과 세는 방법을 하나로 맞춘 작업**이다.
사장님 관점에서 달라진 점:

1. **화면마다 다르던 숫자가 하나로 맞았다.** 같은 '매출'·'미답변 문의'·'재고 위험'이 팀 화면·AI 답변·요약에서 서로 다르게 나오던 것을 공통 계산 규칙(계약)으로 묶었다.
2. **내부 약속어가 사용자에게 새어나가지 않는다.** 카테고리 없는 매출이 `uncategorized`로 보이던 것이 `미분류`로 표시된다.
3. **매출은 이제 "취소·미결제를 뺀 실제 결제금액"이다.** 취소된 주문이 매출에 섞여 부풀던 문제가 사라졌다.
4. **재고 위험은 상품마다 다른 기준으로 본다.** 상품별 안전재고가 있으면 그 값, 없으면 5개를 기준으로 판단한다.
5. **문의는 "미답변"과 "미처리"를 구분한다.** 관리자 확인이 필요한 건과 아직 답을 안 한 건을 더 이상 같은 칸에 넣지 않는다.
6. **모르는 값을 아는 척하지 않는다.** 해석 못 한 문의 상태는 "답변완료"로 뭉개지 않고 별도로 센다.
7. **가장 중요한 변화 — 실제 자료와 시험 자료를 섞지 않는다.** 실제로 연결된 것은 "실제 데이터", 시험용은 "시험 데이터", 연결이 안 되면 "연결 안 됨"으로 표시된다. 연결 실패를 "0건"으로 위장하지 않는다.
8. 이 8가지는 **계산과 표시의 신뢰 기반**이지, 실제 판매 운영이 준비됐다는 뜻은 **아니다**.

## B. 완료 범위 (C-1 ~ C-10)

각 항목의 "대표 검증"은 자동검사(스모크·회귀 하네스)를 뜻하며, 화면 육안검증은 별도로 표기한다.

### C-1. 카테고리 미분류 표시
- **기존 문제**: 카테고리 정보가 없는 매출에 내부 통일 키 `uncategorized`가 **사용자 화면에 그대로 노출**. 숫자가 아니라 표시 계약 위반이라 화면상 성공처럼 보였다.
- **확정 기준**: 분석 카테고리는 `RevenueOrderLine.categoryCode` 하나만 기준. 값 없으면 키는 `uncategorized`로 통일하고 **화면 라벨만 `미분류`**. 하위 엔진이 현재 상품목록으로 재분류하지 않는다(과거 매출의 소급 재분류 방지).
- **사용자 결과**: 카테고리 매출 TOP에 `미분류`가 정상 표기(내부 키 미노출).
- **대표 검증**: 지표 정합성 하네스 T9~T11 (planner·analyticsQueryEngine·scopeInsight 3엔진 동일값: `catLINE=10,000` / `uncategorized=50,000`, `catINDEX`·`catFALLBACK` 부재).
- **Production 반영**: 병합 `951881a` (1차, `--no-ff`).

### C-2. 매출 기준 통일
- **기존 문제**: 이름은 "매출"인데 계산은 취소·미결제까지 포함한 라인합. 화면·AI마다 분모가 달라 비중·성장률·객단가가 함께 틀어졌다.
- **확정 기준**:
  - **매출 = 유효주문(결제완료·미취소) 결제금액** (`operationalRevenue` / 사용자 표기 **운영매출**, 배송비 포함·할인 차감 후).
  - **상품·카테고리·브랜드 = 유효주문 상품 라인 매출** (배송비 제외). 주문축과 라인축을 섞지 않는다.
  - **"순매출"·`netRevenue` 금지** — 환불·정산을 계산하지 않으므로 그렇게 부르지 않는다. 기존 `netOrderRevenue`는 **삭제하지 않고 deprecated 별칭**으로 보존.
- **사용자 결과**: 헤드라인 매출·객단가·비중·성장률·기간비교가 모두 같은 기준. 라벨이 "운영매출(결제완료·미취소)"로 명시.
- **대표 검증**: `smoke-c2-revenue-basis-parity-v0` BASE 9/0 · 계약 17/17 · `smoke-c2-revenue-naming-user-surface-v0` 7/0.
- **Production 반영**: 병합 `cb29c90` (2차).

### C-3. 재고 위험 기준 통일
- **기존 문제**: 같은 재고를 채팅은 ≤5, 스냅샷·대시보드·캘린더는 ≤20으로 판정 → 같은 상품의 위험 건수가 소비자마다 달랐다(계약 5 · chat 4 · snapshot 8).
- **확정 기준**: `stock<=0` → **out_of_stock**, `0<stock<=safetyStock` → **low_stock**, 그 외 **ok**. **상품별 safetyStock 우선, 누락·NaN·음수·잘못된 값이면 전역 기본값 5**(0은 유효한 설정으로 취급). 위험 합계 = out_of_stock + low_stock이되 상태별 수량도 분리 노출. 판정 근거(stock·resolvedSafetyStock·safetyStockSource)를 결과에 남긴다.
- **사용자 결과**: 채팅·대시보드·스냅샷·캘린더가 같은 위험 건수. 위험 사유를 되짚을 수 있다.
- **대표 검증**: `smoke-c3-inventory-risk-contract-v0` BASE 8/0 · 계약 10/10 · `smoke-c3-safetystock-pipeline-v0` 5/0 · `smoke-c3-synthetic-stock-distribution-v0` BASE 4/0 · 계약 7/7(분포 비퇴화).

### C-4. 문의 상태 정규화
- **기존 문제**: 같은 문의 묶음의 "미답변"이 소비자마다 1·5·6·7·8로 갈렸다(영어 literal / 5-alt 정규식 / `!isAnswered` / 한국어 literal). 관리자 확인 필요(needs_human)가 미답변에 섞이기도 했다.
- **확정 기준**:
  - **canonical 6종**: `unanswered · in_progress · on_hold · needs_human · answered · unknown`.
  - **미답변 = unanswered만**, **미처리(unresolved) = answered 외 전부(unknown 포함)** — 두 개념은 다르다.
  - 정규화는 **입력 경계에서 1회**, 소비자는 공통 판정 함수만 사용. `normalizeInquiryStatus`는 **idempotent**.
  - 해석 못 한 값은 **unknown**으로 별도 집계하고 `rawStatus`·`normalizationReason`을 보존(단, 개인정보는 표본에 넣지 않는다).
  - 알려진 별칭만 매핑하며 **추측 매핑 금지**. `답변대기`는 **MOCK/spec 기반 호환 표본**이지 실제 고도몰 관측값이 아니다.
- **사용자 결과**: 분석 채팅·CS 대시보드·총괄 스냅샷의 "미답변"이 같은 값. "미처리"는 미처리대로 유지되고 unknown이 별도로 보인다.
- **대표 검증**: `smoke-c4-inquiry-status-contract-v0` BASE 20/0 · 계약 8/8.
- **Production 반영**: 병합 `cd148c04` (4차, 2026-07-23 push).

## D. 데이터 출처의 현재 진실 (가장 중요한 기록)

> **지금 화면의 큰 숫자는 대부분 시험 자료다.** 이것은 결함이 아니라 현재 단계의 사실이다.

**현재 고도몰은 API 학습·검증용 테스트 쇼핑몰이다.** 실제 판매 몰이 아니다.

실제 API로 확인된 자료(2026-07-22 실측 기준선):
- 실제 상품 **13개**
- 실제 수기 주문 **1건**
- 실제 현재 재고 항목 **13개**(Goods_Search 재고 필드 파생)
- 실제 유효 판매 **0건**

없는 것:
- **실제 고객 없음 · 실제 상업 매출 없음 · 실제 환불 없음 · 실제 문의 없음 · 실제 리뷰 없음**

- 실제 고도몰 **문의 상태 어휘는 아직 한 번도 관측된 바 없다**(`답변대기`는 MOCK/spec 표본).

가상(시험) 자료:
- **2년치 주문·매출·고객·문의·리뷰·재고 변동은 전부 가상 운영자료(시뮬레이션)다.**
