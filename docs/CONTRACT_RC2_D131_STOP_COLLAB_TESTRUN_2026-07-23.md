## 1. 쉬운 말로 정리한 이번 정책

### HQ 중단 요청 ≠ 실제 업무 중단

총괄이 시킨 일이라도, 총괄은 그 팀의 실무 상태를 직접 바꾸지 않습니다.
그만두고 싶으면 **중단 요청**을 보냅니다. 요청을 보내도:

- 업무 상태는 그대로 (진행 중이면 계속 진행 중)
- 수행자·결과물·첨부·수행자 변경 이력 그대로
- 담당 AI 실행을 끊지 않음
- 기존 카드에 "중단 요청 도착 — 총괄 관리자: (사유)" 한 줄이 뜰 뿐

### 실제 중단은 담당 팀장

담당 팀장(또는 명시적으로 지정된 임시 책임자)이 **같은 카드에서** `작업 중단`을 눌러야
실제로 중단됩니다. 이때 사유는 필수입니다. 총괄 사유를 그대로 써도 되고 보충해도 됩니다.
총괄의 요청 시각·사유와 팀장의 처리 시각·사유가 **둘 다** 남습니다.

### 일반 채팅은 자동 상태변경 없음

총괄이 채팅으로 "그 업무 중지해"라고 말할 수 있습니다. 하지만 채팅은

- 업무를 자동으로 찾지 않고
- 상태를 자동으로 바꾸지 않고
- 정식 중단 요청으로 자동 변환하지 않습니다.

그 말은 **총괄 관리자가 담당 팀에 보낸 일반 메시지**로만 남습니다. 담당 팀장이 업무 카드를
찾아 직접 중단하고 사유를 씁니다. 협의가 필요하면 기존 채팅으로 합니다 —
`거부`·`협의 필요` 같은 새 상태나 버튼은 만들지 않았습니다.

### 중단은 삭제가 아니며 기록 보존

중단된 업무에도 결과물·첨부 참조·수행자와 변경 이력·진행 기록·총괄 중단 요청·팀장 처리
사유·기존 결정 기록이 **전부** 남습니다. 중단된 업무를 일반 배정 함수로 되살리지 않습니다.

### 요청 부모는 추적용, 수행 자식은 실행용

팀 간 협업은 요청팀 카드(부모)와 수행팀 카드(자식) 두 건으로 기록합니다. 하지만

- **요청팀 카드는 지켜보는 용도** — 수행자 배정·직접 처리·결과 제출 버튼이 없습니다(서비스에서도 거부)
- **수행팀 카드에서만 실제로 처리**합니다
- 같은 일이 두 팀에서 두 번 실행되지 않습니다

### 사용자 화면은 한 흐름

기록은 두 건이지만 화면에는 한 장으로 보입니다.

| 보는 사람 | 보이는 것 |
|---|---|
| 수행팀장 | 실행 가능한 자식 카드 1장 |
| 요청팀장 | 추적 카드 1장 + "🔗 수행: 디자인팀 · 진행 중 · 담당자" 한 줄 |
| 총괄 | 같은 협업 1흐름 |

자식이 진행되면 부모도 진행 중, 자식이 반송·중단되면 부모도 같은 결말과 사유,
**자식이 정상 완료되면 부모도 완료**됩니다(부모가 영원히 진행 중으로 남지 않습니다).

### 메시지 발신 신원

총괄이 상품팀 화면을 열어 놓고 메시지를 보내도 발신자는 **총괄 관리자**입니다.
선택한 팀은 받는 쪽일 뿐입니다. 메시지 처리(진행중·완료)는 **받은 팀 본인만** 합니다.
총괄의 일반 메시지는 업무를 자동으로 만들지 않고, **팀장 간 명시적 '지원 요청'만**
협업 업무를 만듭니다.

### 현재 운영 시작은 시험 시나리오

지금 `오늘의 운영 시작` 버튼과 채팅 실행은 검증 시나리오(준비된 시험 자료)로 도는
**시험 운영**입니다. 그래서

- 결과를 실제 업무·승인함에 저장하지 않습니다
- 만들어지는 지시 카드에 `[시험 시나리오]` 표식이 붙습니다
- 리포트·이력에 `[시험 운영]` 표시가 붙고, 이력의 출처는 **이 실행이 실제로 쓴 입력**으로
  저장됩니다(화면에 실제 데이터가 열려 있다고 실제로 기록하지 않습니다)
- 실제 업무는 각 팀장에게 지시로만 전달됩니다

시험 시연 자체는 그대로 둡니다. 다만 실제 지시·실제 완료로 기록하지 않습니다.

## 2. 최소 구조 (늘어난 것 전부)

```ts
// 기존 LifecycleTask 에 붙는 선택 필드 2개
stopRequests?: StopRequestRecord[];   // append-only. 요청일 뿐 상태를 바꾸지 않는다
trackingOnly?: boolean;               // 협업 요청팀 카드 = 지켜보는 카드

interface StopRequestRecord {
  requestedBy: ActorRef;
  reason: string;
  requestedAt: string;
}
```

새 저장소·새 상태 머신·새 결재함은 없습니다. 실제 중단 결정은 기존 `decisions` 이력에
팀장 행위로 남습니다.

| GAP | 결함 | 정정 |
|---|---|---|
| A | 총괄이 요청자 자격으로 담당 팀 업무를 즉시 `stopped` 로 만들 수 있었다 | 실제 중단은 담당 팀장만. 총괄은 `requestTaskStop` 으로 요청만 |
